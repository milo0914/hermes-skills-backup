# GRPO Planning 工作流程

## ⚠️ 智能開關：任務分級路由 (Task Routing)

**在執行本技能的任何流程前，必須先進行內部快速掃描（不輸出），判斷任務等級：**

### 快速掃描指標 (Heuristics)

| 指標 | 觸發條件 | 傾向等級 |
|------|----------|----------|
| **關鍵字** | "規劃", "架構", "重構", "複雜", "分析", "設計", "為什麼 (深層)", "debug", "優化" | L2/L3 |
| **上下文長度** | 用戶提供了大量背景資訊、代碼或數據 | L2/L3 |
| **風險評估** | 涉及刪除檔案、修改生產環境、金額交易、安全漏洞 | **強制 L3** |
| **模糊性** | 需求不明確，需要多步推導或假設驗證 | L2/L3 |
| **用戶顯式要求** | "請深度思考"、"用 GRPO 模式" | **強制 L3** |

### 分級處理決策樹

| 等級 | 類型 | 特徵 | **本技能處理策略** | 範例 |
|------|------|------|-------------------|------|
| **L0** | 閒聊/事實檢索 | 問候、簡單事實、標準答案 | **🚫 跳過本技能**，直接回答，禁止輸出思考過程 | "你好"、"Python 是什麼？"、"現在幾點？" |
| **L1** | 常規技能調用 | 單一工具調用、固定 SOP | **🚫 跳過本技能**，直接調用工具，簡要說明意圖 | "查台北天氣"、"存這個檔案" |
| **L2** | 複雜推理/多步驟 | 需分解任務、有依賴關係、中等風險 | **✅ 啟用簡化版 Planning**：列出 3-5 步計劃，逐步執行，**無需群體採樣** | "分析 CSV 趨勢並畫圖"、"重構這個函數" |
| **L3** | 高風險/開放式創新 | 無標準答案、高錯誤成本、需權衡取捨 | **✅ 啟用完整 GRPO 流程**：群體採樣 (3-8 方案)、規則評估、深度反思 | "設計微服務架構"、"修復嚴重 Bug"、"交易日誌策略" |

### 重要規則

1. **預設關閉**：如果無法確定任務等級，預設從 L2 開始，但允許在分析後降級。
2. **動態降級**：即使進入 L3 流程，若發現問題其實很簡單，**立即停止**複雜流程，直接給出答案並告知用戶：「我原本以為這很複雜，但其實很簡單...」。
3. **用戶顯式覆蓋**：若用戶明確要求「請深度思考」或「用 GRPO 模式」，則強制啟動 L3 流程。
4. **禁止過度工程**：對 L0/L1 任務使用完整 GRPO 流程視為**嚴重錯誤**。
5. **效率優先**：思考過程本身不應該成為負擔，若規劃時間超過執行時間，應簡化流程。

## 核心理念

結合 DeepSeek-R1 的 GRPO (Group Relative Policy Optimization) 與 Planning 機制，實現：
1. **Drawing a map before setting out**：在執行任何任務前，先進行深度規劃
2. **Group Sampling**：對關鍵節點生成多個候选方案並比較
3. **Rule-based Reward**：用客觀規則評估方案優劣
4. **Iterative Refinement**：根據獎勵反饋優化策略

## 何時使用

- ✅ 用戶任務複雜度高，需要多步驟規劃
- ✅ 存在多種可能的解決方案，需要比較優劣
- ✅ 有明確定義的成功標準（可自動化驗證）
- ✅ 需要超越直覺反應（System 1），啟用深度推理（System 2）
- ❌ **不要**用於：閒聊、簡單事實查詢、單一工具調用

## 工作流程

```
[用戶提問]
 ↓
[前置檢查] 任務分級路由 (L0-L3) ← 新增
 ↓
IF L0/L1: [直接回答] → 結束
IF L2: [簡化 Planning] → 執行 → 結束
IF L3: [完整 GRPO 流程]
 ↓
[階段 1: 規劃] 生成詳細計劃（使用 writing-plans）
 ↓
[階段 2: 群體採樣] 對關鍵節點生成 G 個候选方案 (G=3-8)
 ↓
[階段 3: 規則評估] 用預定義規則計算每個方案的獎勵分數
 ↓
[階段 4: 選擇與執行] 選擇最高分方案執行
 ↓
[階段 5: 反思] 執行後驗證結果，更新策略
 ↓
[輸出最終答案]
```

## 詳細步驟

### 前置檢查：任務分級 (Task Routing)

**目標**：快速判斷任務等級，決定是否啟用 GRPO 流程

**內部思考模板**（不輸出）：
```
快速掃描：
- 有關鍵字觸發嗎？ → 是/否
- 上下文很長嗎？ → 是/否
- 有高風險嗎？ → 是/否
- 需求模糊嗎？ → 是/否

決策：
- 如果全是「否」→ L0/L1，直接回答
- 如果中等複雜 → L2，簡化 Planning
- 如果高風險/開放式 → L3，完整 GRPO
```

### 階段 1: 規劃（Drawing the Map）

**目標**：在給出最終答案前，先生成完整的推理鏈和執行計劃

**提示詞模板**：
```
請先輸出 <think> 標籤內的思考過程，包括：

1. **任務理解**：用一句話重述用戶的核心需求
2. **任務分解**：將複雜任務拆分為 N 個獨立子任務
3. **風險評估**：列出每個子任務的潛在風險和應對策略
4. **驗證標準**：定義每個子任務的成功標準（必須是可量化的）
5. **備選方案**：對不確定的部分，列出 2-3 個可能的解決方向

然後在 </think> 後輸出實際行動。
```

**關鍵要求**：
- 強制模型在 `<think>` 標籤內完成所有規劃
- 規劃必須包含可量化的驗證標準
- 對不確定的部分必須提供備選方案

### 階段 2: 群體採樣（Group Sampling）

**目標**：對關鍵決策點生成多個候选方案

**實作方式**：
```python
# 對每個關鍵決策點，生成 G 個候选方案
num_generations = 5 # GRPO 中的 G
candidates = []

for i in range(num_generations):
 candidate = generate_solution(
 task=task,
 temperature=0.7 + 0.05 * i, # 溫度遞增增加多樣性
 max_tokens=2048
 )
 candidates.append(candidate)
```

**提示詞模板**：
```
請為以下任務生成 {G} 個不同的解決方案：

任務：{task_description}

要求：
1. 每個方案必須有不同的切入點或策略
2. 方案 A：最保守/直接的方法
3. 方案 B：平衡型方法
4. 方案 C：最創新的方法
5. 方案 D：最小代價方法
6. 方案 E：最易擴展方法

請用 <solution_A>, <solution_B> 等標籤區分每個方案。
```

### 階段 3: 規則評估（Rule-based Reward）

**目標**：用客觀規則評估每個方案，計算獎勵分數

**獎勵函數設計**：
```python
def reward_function(candidate, task_type):
 rewards = {}
 total_score = 0.0

 # 1. 正確性獎勵 (0-1.0)
 if verify_correctness(candidate):
 rewards['correctness'] = 1.0
 else:
 rewards['correctness'] = 0.0

 # 2. 完整性獎勵 (0-0.3)
 completeness = check_completeness(candidate)
 rewards['completeness'] = 0.3 * completeness

 # 3. 效率獎勵 (0-0.2)
 efficiency = estimate_efficiency(candidate)
 rewards['efficiency'] = 0.2 * efficiency

 # 4. 格式獎勵 (0-0.1) - 鼓勵包含思考過程
 if '<think>' in candidate and '</think>' in candidate:
 rewards['format'] = 0.1
 else:
 rewards['format'] = 0.0

 # 5. 自我驗證獎勵 (0-0.2) - 鼓勵自我檢查
 reflection_count = candidate.count('wait, let me verify')
 rewards['reflection'] = 0.05 * min(reflection_count, 4)

 total_score = sum(rewards.values())
 return total_score, rewards
```

**評估提示詞**：
```
請根據以下規則評估每個方案：

評分標準：
1. **正確性** (0-1.0 分)：方案是否正確解決問題？
2. **完整性** (0-0.3 分)：是否覆蓋所有邊界情況？
3. **效率** (0-0.2 分)：時間/空間複雜度如何？
4. **格式** (0-0.1 分)：是否包含完整的思考過程？
5. **自我驗證** (0-0.2 分)：是否有自我檢查和修正？

請為每個方案輸出：
<solution_X>
分數：X.X
優點：...
缺點：...
改進建議：...
</solution_X>
```

### 階段 4: 選擇與執行

**目標**：選擇最高分方案並執行

**提示詞**：
```
根據評估結果，選擇分數最高的方案進行執行。

選擇理由：
- 方案 X 得分最高 (X.X 分)
- 優勢：...
- 已考慮的風險：...

開始執行：
```

### 階段 5: 反思與更新

**目標**：執行後驗證結果，更新策略

**提示詞**：
```
執行完成後，請進行反思：

1. **結果驗證**：實際結果是否符合預期？
2. **偏差分析**：如果有偏差，原因是什么？
3. **策略更新**：下次遇到類似任務，應該如何改進？

反思結果將用於優化未來的決策策略。
```

## 與現有技能集成

### 與 writing-plans 集成
- 在 `writing-plans` 的基礎上，增加群體採樣步驟
- 對計劃中的關鍵決策點進行多方案比較

### 與 subagent-driven-development 集成
- 實現子智能體在開始前進行群體採樣
- 審查子智能體根據規則獎勵進行評估

### 與 test-driven-development 集成
- 將規則獎勵與測試用例結合
- 測試通過作為正確性獎勵的依據

## 關鍵成功要素

1. **智能開關**：正確判斷任務等級，避免過度工程
2. **強制思考**：必須在 `<think>` 標籤內完成規劃才能執行
3. **多樣性**：群體採樣必須有足夠的多樣性（溫度參數調整）
4. **客觀規則**：獎勵函數必須基於可量化的規則
5. **迭代優化**：根據反思結果更新策略
6. **驗證先行**：在執行前定義清晰的驗證標準
7. **動態降級**：發現問題簡單時立即簡化流程

## 注意事項

- ✅ **要**先判斷任務等級再決定是否啟用 GRPO
- ✅ **要**對 L0/L1 任務直接回答，不啟用複雜流程
- ✅ **要**在發現問題簡單時立即降級處理
- ❌ **不要**為了群體採樣而犧牲響應速度
- ❌ **不要**使用主觀獎勵 — 所有規則必須可量化
- ❌ **不要**忽略反思階段 — 這是策略更新的關鍵
- ❌ **不要**對簡單任務使用完整 GRPO 流程
- ✅ **必須**區分思考與執行 — 思考在 `<think>` 標籤內完成
- ✅ **必須**有驗證環節 — 沒有驗證的規劃是無效的
